# Approval Process — دورة اعتماد المصطلحات

## الغرض

توثيق **المسار الرسمي** لإدخال أي مصطلح جديد إلى قاموس AUNAK. لا يُستخدم مصطلح في النظام قبل اكتمال هذه الدورة.

**هذه الوثيقة عملية إجرائية — لا تحتوي على مصطلحات فعلية.**

## المرحلة 1 — اقتراح (Proposal)

### الوصف

يُقدَّم طلب إضافة مصطلح جديد أو استبدال مصطلح ممنوع ببديل رسمي.

### المسؤولية

| الدور | المسؤولية |
|-------|-----------|
| **مقدّم الاقتراح** | أي عضو في الفريق (هندسة، منتج، محتوى، تخصصات) |
| **المخرجات** | مسودة مصطلح مكتملة وفق `TERM_SCHEMA.md` |
| **الحالة** | `proposed` |

### المتطلبات

- تعبئة جميع الحقول الإلزامية في القالب.
- الإشارة إلى **الحاجة** — لماذا يُقترح هذا المصطلح الآن؟
- التحقق الأولي من `NAMING_RULES.md` (لا مترادفات، لا غموض).

### ما لا يحدث في هذه المرحلة

- لا يُستخدم المصطلح في الكود أو Airtable أو الواجهات.
- لا يُنشر كمرجع رسمي.

---

## المرحلة 2 — مراجعة (Review)

### الوصف

فحص المسودة من حيث الاتساق اللغوي والعلمي والهندسي مع الإطار القائم.

### المسؤولية

| الدور | المسؤولية |
|-------|-----------|
| **المراجع** | مسؤول القاموس / Architecture Team (يُحدَّد لاحقاً) |
| **المخرجات** | تقرير مراجعة: قبول، رفض، أو «يحتاج تعديل» |
| **الحالة** | `under_review` |

### معايير المراجعة

- هل الاسم العربي والإنجليزي متسقان؟
- هل التعريف الرسمي والتشغيلي متمايزان وواضحان؟
- هل «الكلمات الممنوعة» تغطي التسميات المتعارضة المعروفة؟
- هل «يستخدم في» واقعي وقابل للتطبيق؟
- هل يتعارض مع مصطلح معتمد آخر (عند وجود قاموس ممتلئ)؟

### المدة

- مراجعة أولية خلال فترة يُحددها الفريق (توصية: 5 أيام عمل).

## المرحلة 3 — مناقشة (Discussion)

### الوصف

نقاش مفتوح للاعتراضات والبدائل قبل القرار النهائي.

### المسؤولية

| الدور | المسؤولية |
|-------|-----------|
| **المشاركون** | أصحاب المصلحة: هندسة، تخصصات، محتوى، QA |
| **المخرجات** | محضر مختصر: قرارات، اعتراضات مُسجَّلة، تعديلات على المسودة |
| **الحالة** | `under_review` |

### قواعد المناقشة

- الاعتراضات **مكتوبة** مع سبب — لا رفض شفهي دون توثيق.
- البدائل تُقيَّم وفق `NAMING_RULES.md`، لا وفق التفضيل الشخصي.
- إن طُلبت تعديلات جوهرية، تعود المسودة إلى مقدّم الاقتراح ثم تُعاد المراجعة.

---

## المرحلة 4 — اعتماد (Approval)

### الوصف

قرار رسمي بقبول المصطلح كمرجع معتمد.

### المسؤولية

| الدور | المسؤولية |
|-------|-----------|
| **المعتمد** | Architecture Team / مالك القاموس (يُحدَّد لاحقاً) |
| **المخرجات** | TERM-ID نهائي، إصدار `1.0.0`، تاريخ اعتماد |
| **الحالة** | `approved` |

### شروط الاعتماد

- اكتمال المراجعتين (مراجعة + مناقشة) دون اعتراضات معلّقة.
- توافق مع الدستور و`002_UNIFIED_TERMINOLOGY_PHILOSOPHY.md`.
- عدم وجود مصطلح معتمد يغطي نفس المفهوم (اسم واحد لكل مفهوم).

---

## المرحلة 5 — إضافة للقاموس (Registry Entry)

### الوصف

نشر المصطلح المعتمد في موقعه الرسمي داخل `docs/terminology/`.

### المسؤولية

| الدور | المسؤولية |
|-------|-----------|
| **مسؤول القاموس** | إدراج الملف، فهرسته، ربط TERM-ID |
| **المخرجات** | ملف مصطلح بحالة `approved` + تحديث الفهرس (عند إنشائه) |

### المتطلبات

- الملف يطابق `TERM_SCHEMA.md` بالكامل.
- لا تُحذف مسودات الاقتراح — تُؤرشف أو تُربط بالإصدار النهائي.

---

## المرحلة 6 — إتاحته للاستخدام (Availability)

### الوصف

يصبح المصطلح **مسموحاً** في الطبقات التنفيذية والوثائق التقنية.

### المسؤولية

| الدور | المسؤولية |
|-------|-----------|
| **فرق التنفيذ** | استخدام المصطلح في DB / Code / API / UI / AI حسب «يستخدم في» |
| **مسؤول القاموس** | إعلان الإتاحة للفريق (قناة داخلية، changelog مصطلحات) |

### قواعد الإتاحة

- **قبل** هذه المرحلة: استخدام المصطلح في أي طبقة **مخالفة للإطار**.
- **بعد** هذه المرحلة: أي استخدام باسم مختلف لنفس المفهوم **مخالفة**.
- التعديلات اللاحقة تخضع لـ `CHANGE_POLICY.md` — لا تغيير صامت.

## حالات خاصة

| الحالة | المسار |
|--------|--------|
| **رفض الاقتراح** | يُوثَّق السبب؛ لا يُدرج في القاموس |
| **دمج مقترحين لنفس المفهوم** | مناقشة واحدة → مصطلح واحد → TERM-ID واحد |
| **استعجال (hotfix لغوي)** | يمرّ بنفس المراحل؛ قد تُختصر المناقشة بموافقة صريحة موثّقة |
| **إهمال مصطلح قديم** | مسار منفصل عبر `CHANGE_POLICY.md` (deprecated → retired) |
