# نظام إدارة المعرفة وتتبّعها (Knowledge Traceability System)

## الهدف منها

تصميم نظام موحّد لإدارة وتتبّع جميع الوثائق الفكرية والهندسية داخل مشروع AUNAK، بحيث تصبح كل فكرة، وكل قرار، وكل جلسة، وكل وثيقة قابلة للتتبع والربط مع بقية مكونات المشروع.

**حالة الوثيقة:** Draft  
**الإصدار:** 0.1  
**تاريخ الإنشاء:** 2026-07-06  
**المؤلف:** AUNAK Architecture Team  
**النوع:** وثيقة تصميم — غير مُطبَّقة

---

## الفصل الأول: لماذا يحتاج مشروع طويل العمر إلى نظام تتبع للمعرفة؟

مشروع AUNAK مصمَّم ليعمل على مدى سنوات، وقد يمرّ بفرق متعددة، وتغيّرات تقنية، وتوسعات وظيفية. في مثل هذه البيئة، تتعرّض المعرفة لثلاثة مخاطر أساسية:

1. **الفقدان:** تبقى الأفكار التأسيسية في المحادثات أو الذاكرة الشخصية، فتضيع عند مغادرة أي عضو للفريق.
2. **التشتت:** تتوزع الوثائق بين مجلدات وصيغ وأسماء غير موحّدة، فيصعب معرفة أين وُثّق ماذا.
3. **انقطاع السياق:** يُنفَّذ قرار أو كود دون إمكانية الرجوع إلى السؤال الأصلي أو النقاش الذي سبقه.

نظام تتبّع المعرفة يُعالج هذه المخاطر عبر ربط كل عنصر — فكرة، جلسة، قرار، وثيقة، وحدة برمجية — بمعرّف موحّد وعلاقات واضحة. الهدف ليس زيادة عدد الوثائق، بل **جعل أصل أي فكرة قابلاً للوصول خلال ثوانٍ** مهما مرّ الزمن أو تغيّر الفريق.

## الفصل الثاني: ما المقصود بـ Traceability داخل AUNAK؟

**Traceability** (قابلية التتبّع) تعني القدرة على تتبّع أي عنصر في المشروع — من الفكرة إلى التنفيذ — عبر سلسلة روابط موثّقة، بحيث يمكن الإجابة على الأسئلة التالية في أي وقت:

- من أين جاءت هذه الفكرة؟
- ما القرار الذي اعتمدها؟
- ما الوثائق التي تأثرت بها؟
- ما الكود أو قاعدة البيانات التي نفّذتها؟

### لماذا لا يقتصر على الكود فقط؟

تتبّع الكود عبر Git يجيب على: **ماذا تغيّر ومتى؟**  
تتبّع المعرفة في AUNAK يجيب على: **لماذا بُني بهذا الشكل؟**

الكود وحده لا يحفظ:

- الأسئلة التي سبقت القرار.
- البدائل التي رُفضت.
- الاعتراضات التي أُثيرت.
- المبادئ الفلسفية التي حكمت الاختيار.

لذلك، Traceability في AUNAK يشمل **الطبقة الفكرية** (Philosophy) و**الطبقة المعمارية** (Architecture) و**طبقة القرار** (Decision Log) و**طبقة التنفيذ** (Code, Database, API, AI) — كسلسلة واحدة قابلة للتتبّع، لا كطبقات منفصلة.

## الفصل الثالث: تصميم نظام معرفات موحد

يُقترح نظام معرفات (ID Scheme) موحّد لجميع عناصر المعرفة في المشروع. **هذا تصميم مقترح فقط — لم يُطبَّق بعد.**

### البادئة والرقم

كل معرّف يتكوّن من: **بادئة النوع + رقم تسلسلي من ثلاث خانات**

| البادئة | النوع | المثال | الوصف |
|---------|-------|--------|-------|
| **P-** | Philosophy | P-001 | وثيقة فلسفية أو جلسة تفكير |
| **A-** | Architecture Decision | A-001 | قرار معماري معتمد |
| **S-** | Session | S-001 | جلسة نقاش أو تصميم موثّقة |
| **R-** | Research | R-001 | بحث أو مرجع علمي |
| **M-** | Module | M-001 | وحدة برمجية أو نظام فرعي |
| **DB-** | Database | DB-001 | قرار أو تصميم متعلق بقاعدة البيانات |
| **API-** | API | API-001 | واجهة برمجية أو عقد API |
| **AI-** | AI Component | AI-001 | مكوّن ذكاء اصطناعي |
| **DOC-** | Documentation | DOC-001 | وثيقة هندسية عامة (Identity, Roadmap, إلخ) |

### قواعد مقترحة

1. **التسلسل مستقل لكل بادئة:** A-001 و A-002 قراران مختلفان؛ P-001 و P-002 وثيقتان فلسفيتان مختلفتان.
2. **لا إعادة استخدام:** المعرّف يُخصَّص مرة واحدة ولا يُعاد تعيينه حتى لو حُذفت الوثيقة (يُؤرشف بدلاً من الحذف).
3. **الربط عبر المراجع:** أي وثيقة يمكن أن تحتوي حقل `Related IDs:` ي enumerates المعرّفات المرتبطة.
4. **التوافق مع الموجود:** القرار A-001 (Child Journey Core) المسجّل في `docs/05_DECISIONS_LOG.md` يُعتبر أول تطبيق فعلي لهذا النظام، ويُستخدم كمرجع للتوحيد لاحقاً.

### تحسين مقترح (غير مُطبَّق)

إضافة بادئة **EVT-** (Event) للأحداث الزمنية الكبرى في تاريخ المشروع (مثل: إطلاق MVP، اعتماد قرار سيادي)، لربط سلسلة من القرارات والوثائق بحدث واحد.

## الفصل الرابع: كيف ترتبط الوثائق ببعضها؟

الوثائق في AUNAK ليست جزراً معزولة؛ بل شبكة مترابطة. العلاقة بين أنواع الوثائق كالتالي:

```
                    ┌─────────────┐
                    │  Identity   │  ← الهوية والمبادئ الثابتة
                    │  (DOC-000)  │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │Philosophy│ │ Roadmap  │ │ Research │
        │  (P-xxx) │ │ (DOC-xxx)│ │  (R-xxx) │
        └────┬─────┘ └────┬─────┘ └────┬─────┘
             │            │            │
             └────────────┼────────────┘
                          ▼
                   ┌─────────────┐
                   │  Sessions   │  ← جلسات النقاش
                   │   (S-xxx)   │
                   └──────┬──────┘
                          ▼
                   ┌─────────────┐
                   │  Decisions  │  ← قرارات معتمدة
                   │   (A-xxx)   │
                   └──────┬──────┘
                          │
         ┌────────────────┼────────────────┐
         ▼                ▼                ▼
   ┌───────────┐   ┌───────────┐   ┌───────────┐
   │Architecture│   │  Modules  │   │    AI     │
   │  (DOC-xxx) │   │  (M-xxx)  │   │ (AI-xxx)  │
   └─────┬─────┘   └─────┬─────┘   └─────┬─────┘
         │               │               │
         └───────────────┼───────────────┘
                         ▼
              ┌─────────────────────┐
              │ Database │ API     │  ← التنفيذ
              │ (DB-xxx) │(API-xxx)│
              └─────────────────────┘
```

### جدول العلاقات

| من | إلى | نوع العلاقة |
|----|-----|-------------|
| **Identity** | Philosophy, Roadmap | يُغذّي — المبادئ الثابتة تحكم الفلسفة والخطة |
| **Philosophy** | Sessions | يُولّد — الفلسفة تطرح أسئلة تُناقش في الجلسات |
| **Sessions** | Decisions | يُنتج — الجلسات تُفضي إلى قرارات |
| **Decisions** | Architecture, Modules, DB, API, AI | يُوجّه — القرار يحدّد ما يُبنى |
| **Architecture** | Modules, DB, API | يُحدّد — المعمارية تحدد حدود الأنظمة |
| **Modules** | Database, API | يُنفّذ — الوحدات تُطبَّق في البيانات والواجهات |
| **Research** | Philosophy, Decisions | يُدعم — البحث يقدّم أدلة للفلسفة والقرار |
| **Roadmap** | Decisions, Architecture | يُخطّط — الخطة تحدّد أولويات القرارات والبناء |

## الفصل الخامس: دورة حياة الفكرة داخل المشروع

منذ ولادتها حتى تتحول إلى كود، تمرّ كل فكرة تأسيسية بالمراحل التالية:

```
Idea (فكرة)
        ↓
Discussion (نقاش — Session S-xxx)
        ↓
Philosophy (توثيق فلسفي — P-xxx)
        ↓
Architecture (تصميم معماري — DOC-xxx)
        ↓
Decision (قرار معتمد — A-xxx)
        ↓
Implementation (تنفيذ — M-xxx, DB-xxx, API-xxx, AI-xxx)
        ↓
Validation (تحقق — اختبار، مراجعة)
        ↓
Documentation Update (تحديث الوثائق)
```

### ملاحظات على الدورة

- **ليست خطية دائماً:** قد تعود فكرة إلى مرحلة Philosophy بعد Validation إذا ظهرت نتائج تستدعي إعادة النظر.
- **كل مرحلة تُوثَّق:** لا تنتقل فكرة إلى المرحلة التالية دون توثيق المرحلة الحالية.
- **القرار (Decision) هو نقطة التحوّل:** قبل A-xxx تبقى الفكرة قابلة للنقاش؛ بعد A-xxx تصبح ملزمة للتنفيذ.
- **Documentation Update ليست نهاية:** بعد التنفيذ، تُحدَّث الوثائق الهندسية (`docs/`) لتعكس الواقع الفعلي.

## الفصل السادس: ما فوائد هذا النظام بعد خمس أو عشر سنوات؟

| الفائدة | الوصف |
|---------|-------|
| **نقل المعرفة** | أي مهندس جديد يمكنه تتبّع أصل أي قرار أو مكوّن خلال دقائق، دون الاعتماد على «من كان موجوداً وقتها». |
| **البحث العلمي** | السجل الفكري الكامل (Philosophy + Research + Decisions) يُستخدم كأساس لأ papers ودراسات مبنية على المشروع. |
| **مراجعة القرارات** | عند الحاجة لإعادة تقييم قرار قديم، يمكن الرجوع إلى الجلسة والبدائل والاعتراضات التي سبقته. |
| **تدريب المهندسين الجدد** | المكتبة الفلسفية + سجل القرارات + المعمارية = منهج تدريبي متكامل لفهم «لماذا عونك بُنيت هكذا». |
| **الحفاظ على الهوية الفكرية** | حتى مع تغيّر التقنيات والأفراد، تبقى المبادئ والاستدلالات محفوظة وقابلة للمراجعة. |

## الفصل السابع: ما الذي لا يقوم به هذا النظام؟

توضيحاً لحدود النظام المقترح:

| لا يقوم بـ | السبب |
|-----------|-------|
| **لا يحل محل Git** | Git يتتبّع تغيّرات الكود؛ هذا النظام يتتبّع تغيّرات الفكر والقرار. |
| **لا يحل محل التوثيق البرمجي** | تعليقات الكود وJSDoc وREADME للمكوّنات تبقى مسؤولية المطوّر. |
| **لا يحل محل إدارة الإصدارات** | semver وrelease notes وchangelog تبقى آلية منفصلة. |
| **لا يفرض أداة واحدة** | النظام تصميم مفاهيمي؛ يمكن تطبيقه عبر Markdown + Git، أو أداة خارجية لاحقاً. |
| **لا يضمن جودة المحتوى** | يضمن **وجود** التوثيق و**ربطه**؛ جودة المحتوى مسؤولية الفريق. |

## الفصل الثامن: توصيات مستقبلية (دون تنفيذ)

1. **إضافة حقل `Related IDs` في كل وثيقة** — سطر واحد في ترويسة كل ملف يربطه بالمعرّفات ذات الصلة.
2. **إنشاء فهرس مركزي** — ملف `docs/INDEX.md` يجمع كل المعرّفات مع روابطها (P, A, S, R, M, DB, API, AI, DOC).
3. **ربط Decision Log بـ Philosophy** — عند تسجيل قرار A-xxx، إلزام الإشارة إلى وثيقة P-xxx أو S-xxx التي سبقته.
4. **توحيد A-001 الموجود** — القرار Child Journey Core المسجّل حالياً يُعتبر المرجع الأول؛ توثيق جلسة S-001 ووثيقة P-002 (لماذا الطفل رحلة) كسياق له.
5. **مراجعة دورية** — كل ربع سنة، مراجعة الفهرس للتأكد من عدم وجود وثائق «يتيمة» بلا معرّف أو ربط.
6. **تقييم أداة خارجية** — عند نمو المشروع، تقييم أدوات مثل Notion أو Obsidian أو wiki مخصص لإدارة الروابط بين الوثائق، مع الإبقاء على Git كمصدر الحقيقة.
