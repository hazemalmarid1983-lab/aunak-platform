# Change Policy — سياسة تعديل المصطلحات

## الغرض

تحديد **متى وكيف** يُسمح بتعديل مصطلح معتمد في قاموس AUNAK، وما أثر ذلك على النظام والوثائق، وكيف يُحافَظ على التوافق مع المراجع السابقة.

**هذه الوثيقة سياسة فقط — لا تحتوي على مصطلحات فعلية.**

## أنواع التغيير

| النوع | الوصف | مثال على النمط (إرشادي) |
|-------|-------|-------------------------|
| **Patch** | توضيح نصي دون تغيير معنى | تصحيح صياغة في التعريف الرسمي |
| **Minor** | توسيع حدود طفيف أو إضافة «يستخدم في» | إضافة طبقة UI دون تغيير الاسم |
| **Major** | تغيير الاسم أو المعنى الجوهري | إعادة تسمية كيان يمس DB وAPI |
| **Lifecycle** | deprecated → retired | إيقاف مصطلح لصالح بديل |

---

## متى يُسمح بالتعديل

### يُسمح (Patch / Minor)

- تصحيح أخطاء إملائية أو نحوية **دون** تغيير المعنى.
- إضافة «كلمات ممنوعة» عند اكتشاف تسرّب لغوي.
- توسيع «يستخدم في» أو «مرتبط بـ» عند توسّع النظام.
- تحديث «ملاحظات» أو مراجع علمية.
- رفع إصدار Minor بعد مراجعة مسؤول القاموس.

### يُسمح (Major) — بقرار رسمي فقط

- تغيير الاسم العربي أو الإنجليزي المعتمد.
- تغيير التعريف التشغيلي بما **يغيّر** سلوك النظام أو نطاق الكيان.
- دمج مصطلحين كانا يمثلان مفهوماً واحداً.
- تقسيم مصطلح واحد إلى مصطلحين منفصلين.

**Major يتطلب:** مساراً يعادل `APPROVAL_PROCESS.md` (مراجعة + مناقشة + اعتماد) + خطة ترحيل موثّقة.

### يُسمح (Lifecycle)

- **deprecated:** المصطلح لا يُستخدم في تنفيذ جديد؛ يبقى في القاموس للمرجعية.
- **retired:** يُزال من الاستخدام النشط؛ يُحتفظ بالملف مع TERM-ID للأرشيف.

## متى يُمنع التعديل

| المنع | السبب |
|-------|-------|
| **تعديل صامت** | تغيير الاسم أو المعنى دون مسار اعتماد |
| **تجاوز TERM-ID** | إنشاء ملف جديد بمعرف مختلف لنفس المفهوم دون retired للقديم |
| **تراجع convenience** | تغيير المصطلح لأن الكود «أسهل» باسم آخر |
| **مزامنة عكسية** | تعديل القاموس ليتوافق مع كود غير معتمد — الكود يُصحَّح لا القاموس |
| **حذف TERM-ID** | المعرفات **لا تُحذف** — تُؤرشف فقط |

## أثر تعديل المصطلح

### Patch

| الطبقة | الأثر |
|--------|-------|
| DB / Code / API | لا تغيير مطلوب |
| الوثائق | تحديث اختياري للصياغة |
| AI | لا أثر |

### Minor

| الطبقة | الأثر |
|--------|-------|
| DB / Code / API | قد يتطلب توسيعاً — لا إعادة تسمية |
| الوثائق | تحديث «يستخدم في» |
| AI | مراجعة prompts إن لزم |

### Major

| الطبقة | الأثر |
|--------|-------|
| **DB** | migration أو alias fields — خارج نطاق هذه الوثيقة |
| **Code** | refactor أسماء Models/fields/routes |
| **API** | versioning أو breaking change موثّق |
| **UI** | تحديث جميع التسميات |
| **AI** | إعادة تدريب/ضبط prompts |
| **التقارير** | إعادة تعريف أعمدة ومؤشرات |
| **الوثائق** | تحديث شامل + إشارة للإصدار السابق |

## كيفية توثيق التغيير

كل تعديل معتمد يُسجَّل في **سجل تغييرات المصطلح** (يُنشأ لاحقاً) بالحد الأدنى:

```markdown
## TERM-XXX — [الاسم العربي / English name]

| الحقل | القيمة |
|-------|--------|
| **نوع التغيير** | patch | minor | major | lifecycle |
| **الإصدار السابق** | x.y.z |
| **الإصدار الجديد** | x.y.z |
| **تاريخ السريان** | YYYY-MM-DD |
| **السبب** | … |
| **قرار الاعتماد** | رابط أو مرجع |
| **أثر على التنفيذ** | none | partial | full migration |
| **بديل (إن وُجد)** | TERM-YYY |
```

داخل ملف المصطلح نفسه:

- تحديث **الإصدار** وفق semver.
- إضافة فقرة في **ملاحظات**: «تغييرات الإصدار x.y.z».
- عند Major: الإبقاء على **الاسم السابق** في «الكلمات الممنوعة» أو قسم «Legacy».

## الحفاظ على التوافق مع الوثائق السابقة

### مبدأ «لا محو للتاريخ»

- الوثائق والـ commits القديمة **لا تُعدَّل retroactively**.
- القاموس الحالي يكون المرجع؛ الوثائق القديمة تُربط بإصدار المصطلح أو TERM-ID.

### آليات التوافق

| الآلية | الاستخدام |
|--------|-----------|
| **TERM-ID ثابت** | المرجع عبر الزمن حتى عند تغيير الاسم |
| **deprecated + see TERM-YYY** | يُبقي المعنى واضحاً أثناء الترحيل |
| **جدول legacy** | في ملاحظات المصطلح: «كان يُسمّى سابقاً: …» |
| **إصدار semver** | الوثائق التقنية تذكر «TERM-XXX v2.0.0+» عند الحاجة |

### فترة الترحيل (Major)

1. إعلان `deprecated` للمصطلح/الاسم القديم.
2. فترة coexistence يُحددها الفريق (توصية: sprint واحد على الأقل).
3. تحديث التنفيذ والوثائق النشطة.
4. انتقال القديم إلى `retired`.

## من يملك قرار التغيير

| نوع التغيير | صاحب القرار |
|-------------|-------------|
| Patch | مسؤول القاموس |
| Minor | مسؤول القاموس + إشعار Architecture Team |
| Major | Architecture Team + مسار اعتماد كامل |
| Lifecycle | Architecture Team |

---

## علاقة التغيير بالتنفيذ البرمجي

- **لا يُعدَّل** الكود أو DB **قبل** تحديث القاموس في تغييرات Major.
- **يُعدَّل** الكود **بعد** سريان الإصدار الجديد — في نفس PR أو PR متتابع موثّق.
- Pull Request يستخدم مصطلحاً غير معتمد أو deprecated **يُرفض** عند تطبيق CI لاحقاً.
